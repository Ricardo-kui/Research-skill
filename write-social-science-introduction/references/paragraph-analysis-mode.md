# Paragraph Analysis Mode

Load this reference when the user provides an exemplar paragraph or asks to learn an author's paragraph logic.

This mode is for diagnosis and transfer, not default funnel drafting.

## Use Cases

Use this mode when the user asks to:

- learn from a sample paragraph
- study an author's support-sentence narration
- label sentence functions
- map an introduction's front-end jobs across multiple paragraphs
- extract reusable paragraph rules
- transfer a paragraph logic into new prose

## Core Rule

Diagnose before imitating. Do not jump from admiration to imitation without first making the paragraph's internal jobs explicit.

Treat a named author as an optional case overlay, not as the default template for paragraph design.

## Workflow

1. Identify the paragraph's local job.
2. Segment the paragraph into:
   - topic sentence
   - optional elaboration
   - support
   - analysis
   - transition
3. For each support sentence, assign one primary function:
   - anchor a construct
   - differentiate constructs
   - sequence reasons
   - provide illustration
   - convert evidence into implication
4. Diagnose the sequencing pattern.
5. If the sample spans multiple introduction paragraphs, add an intro-level function map:
   - hook
   - opening paraphrase
   - literature synthesis
   - gap
   - gap-value sentence
   - paper move
   - setting or design justification
   - contribution preview
   - roadmap
6. Separate generic paragraph rules from sample-specific observations.
7. State the evidence boundary of any author-style inference.
8. Extract 3-5 reusable rules.
9. Only then rewrite or imitate if the user asks.

## Evidence Discipline

- `1 sample`
  Extract only provisional local patterns from the provided paragraph. Do not call them the author's stable style.
- `2-3 samples`
  You may infer tentative recurring preferences, but label them as preliminary.
- `4+ varied samples`
  You may describe stronger author-specific tendencies, but still avoid treating them as laws.

Default to the weakest justified claim.

## Output Shape

Return a compact diagnosis with these headings:

- `段落任务`
- `引言功能图` when the sample spans multiple paragraphs
- `句子功能标注`
- `支撑句功能`
- `排序模式`
- `可迁移规则`
- `证据边界`
- `如需模仿应保留什么`
- `如需模仿应避免什么`

If the user asks for a rewrite after the diagnosis, add:

- `迁移写法示例`

## Diagnosis Rules

- Make sentence roles explicit.
- Prefer functional labels over loose paraphrase.
- Distinguish support from analysis; do not collapse them.
- Distinguish local paragraph job from document-level job.
- Distinguish a hook from its opening paraphrase. A vivid first sentence is not yet an argument until the next sentence translates it into the paper's mechanism, question, or evaluative tension.
- If one sentence performs two jobs, name the primary one and note the secondary role briefly.
- Distinguish generic paragraph logic from author-specific preferences.
- Keep author-style claims proportional to the sample size.

## Transfer Rules

Good transfer rules are procedural, such as:

- `Open with the paragraph's claim, not the topic area.`
- `Use the second sentence to narrow the distinction before adding citations.`
- `Let each support sentence answer the same local reader question.`
- `Add one interpretive sentence before the transition.`

Weak transfer rules are merely impressionistic, such as:

- `Sound more sophisticated.`
- `Use more literature.`
- `Be more like Pollock.`

## Failure Modes

- Retelling the paragraph without labeling sentence functions
- Confusing citations with support logic
- Treating all middle sentences as one undifferentiated support block
- Jumping to imitation before extracting reusable rules
- Copying surface phrasing instead of structural logic
- Overgeneralizing from one sample into a stable authorial template
